# AGENTS.md — R&D Knowledge Map (kmap)

Проект команды для трека **«Научный клубок»** хакатона «Норникель AI Science Hack» (июль 2026).

## Техническое задание (суть)

Единая карта знаний R&D горно-металлургической отрасли: система связывает публикации, отчёты, эксперименты, материалы, процессы, оборудование, свойства, экспертов и выводы; отвечает на сложные технические вопросы вида «что уже делали по X при режиме Y и какой эффект на Z», показывает доказательную базу (provenance до цитаты), противоречия источников, пробелы в данных и релевантных экспертов. Жёсткие требования: **безошибочное извлечение чисел/единиц/диапазонов**, многопараметрические запросы (материал+процесс+условия+география+время), верификация фактов, RU/EN, самостоятельное формирование новых связей при пополнении базы. Ограничения хакатона: только открытые LLM, доступные из РФ (OpenAI/Anthropic запрещены); ценится экономия ресурсов (меньше модель — лучше). Полный разбор: [.agents/01-brief.md](.agents/01-brief.md).

## Выбранное решение

**«Проверяемое числовое ядро с эпистемическим слоем»** — выбрано по итогам проверки шести гипотез (журнал экспериментов: [.agents/02-hypotheses.md](.agents/02-hypotheses.md)):

- числа, единицы, операторы извлекаются **детерминированно** (numcore + реестр единиц + SI + sanity-проверки), LLM физически не может породить число в ответе (numeric guard) — закрывает главное «недопустимо» ТЗ;
- LLM (открытые Qwen3 через OpenAI-совместимый API или vLLM on-prem) — только для сущностей/связей/привязки/синтеза/судьи противоречий, всегда под JSON Schema с пост-верификацией;
- эпистемический слой: консенсус, противоречия (candidate → LLM-judge → эксперт; сравнение только при сопоставимых условиях), heatmap покрытия и пробелов, профили экспертов;
- граф — модель данных (таблица рёбер + рекурсивные CTE + ego-визуализация), «институциональная память»: эксперт → отчёты → эксперименты;
- один стор — **PostgreSQL 18 + pgvector** (вектор+FTS+граф+JSONB+реляционка, ACID); отдельные Neo4j/Qdrant отклонены замерами, пороги пересмотра зафиксированы;
- прод-контур: **микросервисы на Go 1.25 — один Go-проект** (единый `go.mod`, точки входа в `cmd/<service>`, bounded contexts в `internal/`, один Dockerfile с `ARG SERVICE`), Python для ML-сервисов (Docling-парсинг, извлечение, эмбеддинги bge-m3), **gRPC** внутри, **NATS JetStream** для конвейера, **REST/OpenAPI 3.1 через swag (swaggo) v2** наружу, MinIO, OTel/Prometheus/Grafana; отказоустойчивость: outbox, DLQ, идемпотентность, graceful degradation;
- **управление доступом по ТЗ**: OIDC (Keycloak — прод; `auth.mode: demo` со статическими токенами-ролями — демо), RBAC (роли ТЗ + эксперт-валидатор), уровни доступа документов через RLS в PostgreSQL, полный аудит действий ([.agents/03-architecture.md](.agents/03-architecture.md) ADR-6);
- деплой: **Docker Compose на одной Ubuntu VM (8 vCPU / 32 ГБ), без Kubernetes** — enterprise-дисциплина без оркестратора (healthchecks, лимиты, TLS, офф-VM бэкапы, деплой с откатом); LLM-инференс на VM не размещается (внешний upstream); путь масштабирования зафиксирован в [.agents/12-deployment.md](.agents/12-deployment.md) §7.

## Где детали реализации

**Все конкретные детали — в [.agents/](.agents/README.md)** (оглавление там же): требования и ограничения (01), гипотезы и эксперименты (02), архитектура и ADR (03), модель данных и DDL (04), конвейер ingestion (05), извлечение (06), запрос→ответ (07), эпистемика (08), спецификации сервисов (09), контракты REST/gRPC/NATS (10), НФТ и безопасность (11), деплой и CI/CD (12), метрики качества (13), план работ (14), ресурсы (15), стандарты кода и dev-окружение (16), фронтенд и дизайн-система (17).

Документы в `.agents/` — источник истины. Код следует им; осознанные отклонения фиксируются правкой соответствующего документа в том же PR.

## Правила для агентов и участников

1. **Временные файлы — только в `.tmp/`** (черновики, дампы, промежуточные выгрузки, результаты прогонов); **исходные данные хакатона — только в `data-sources/`** (корпус, каталоги, справочники). Обе папки в .gitignore; не складывать данные и мусор в корень или сервисы.
2. **Недостающие ресурсы и доступы — запрашивать у владельца проекта явно** (что, зачем, в каком виде), а не обходить хардкодом. Владелец предоставляет по запросу; **LLM API уже выдан** — DigitalOcean Gradient (OpenAI-совместимый; chat + embeddings + rerank проверены), ключ — в `configs/secrets.yml` (в git не коммитится); доступные модели, цены, ограничения (только open-weight!) и результаты проверки — [.agents/15-resources.md](.agents/15-resources.md) §1. Реестр остальных ресурсов и открытых вопросов — там же.
3. Инварианты проекта (см. [.agents/README.md](.agents/README.md)): детерминированные числа + guard; provenance у каждого факта; факты не удаляются (supersede); один PostgreSQL; контракты только аддитивно; секреты не коммитятся.
4. Стек зафиксирован: Go 1.25.x — **в `go.mod` строго `go 1.25.0`**, весь Go-бэкенд — **один проект: точки входа в `cmd/<service>/main.go`, контексты в `internal/`** (чистая архитектура, DDD, gRPC+buf, swag v2/OpenAPI 3.1, chi; контексты не импортируют друг друга — только `internal/platform` и `contracts/`); Python 3.13 (Docling, bge-m3, numcore); PostgreSQL 18 + pgvector 0.8, NATS JetStream ≥2.12, MinIO, Keycloak (прод; демо — `auth.mode: demo`), OTel. **Вся конфигурация — только YAML** в `configs/` (`base/` + оверлеи `dev|demo|prod` + `configs/secrets.yml` для ключей — он в .gitignore, шаблон `secrets.yml.example`); env-переменные и `.env` не используются вовсе; никаких настроек, разбросанных по флагам/коду. **YAML-файлы всегда именуются с расширением `.yml`** (не `.yaml`). Отклонения — через ADR в [.agents/03-architecture.md](.agents/03-architecture.md).
5. **Качество кода — обязательное.** Go: **golangci-lint с включённым линтером `modernize`** (правила `go fix`/gopls) — локально `golangci-lint run --fix`, в CI без `--fix`; Python: ruff + mypy. Строгое разделение по папкам и слоям и в Go, и в Python (`domain` / `app` / `ports` / `adapters`, эталоны — [.agents/03-architecture.md](.agents/03-architecture.md) §9 и [.agents/16-dev-environment.md](.agents/16-dev-environment.md)). Docker-образы — **на базе alpine, когда это возможно** (Go/UI — всегда; Python ML — `slim`).
   **Комментарии в коде запрещены — в YAML-файлах тоже.** Код и конфиги должны быть самодокументируемыми: понятные имена, структура, ключи; пояснения живут в `.agents/`, а не в исходниках. Запрет распространяется на Go, Python, TypeScript, SQL, YAML (конфиги, compose, CI-воркфлоу).
6. **Не переизобретать велосипед.** Перед самописной реализацией нужно проверить, есть ли зрелый инструмент, библиотека, протокол, генератор или проверенный open-source компонент, который закрывает задачу проекта. При сопоставимых рисках выбирается готовое решение; самописный код допустим только когда он проще, уже является тонкой glue-логикой вокруг выбранных библиотек или есть понятное ограничение по ТЗ/архитектуре. Для повторяющихся задач запрещено писать «маленький свой фреймворк»: HTTP routing, REST↔gRPC transcoding, validation, retry/backoff, config merge, SQL-access, migrations, mocks, codegen, auth, metrics/tracing, S3/NATS/PG-клиенты, таблицы/графы/UI-компоненты берутся из зрелых библиотек или генераторов. При добавлении новой зависимости или dev/runtime-инструмента обязательно фиксировать **что это такое, зачем оно нужно, почему выбрано, где используется и какие есть ограничения**: в релевантном документе `.agents/` (архитектура, dev-окружение, контракты, деплой, фронтенд и т.д.) либо в ADR, если выбор влияет на архитектуру. Если готовое решение сознательно не используется, причина фиксируется там же.
7. PR-гейты: golden-тесты numcore и линтеры зелёные; изменения промптов/моделей — с прогоном eval ([.agents/13-evaluation.md](.agents/13-evaluation.md)).
8. Для работы с GitHub используем утилиту **`gh`** (репозиторий, PR, secrets, Actions). Полный список обязательных dev-утилит, версии и команды установки — [.agents/16-dev-environment.md](.agents/16-dev-environment.md); раннер задач — Taskfile (`task <target>`).
9. **Коммиты — в стиле Conventional Commits, описание на русском языке**: `<type>: <описание по-русски>` (типы: `feat`, `fix`, `docs`, `refactor`, `test`, `perf`, `ci`, `chore`). Пример:
   - `feat: добавлен метод для взаимодействия с LLM`
   Сообщение — одна строка, **без трейлеров** (`Co-Authored-By`, `Generated-By` и т.п. не добавлять никогда). **Порядок всегда: `git pull` → `git commit` → `git push`** — pull делается непосредственно перед коммитом (в репо работают параллельно), незапушенных локальных коммитов не оставляем. **Коммит — после каждого завершённого логического шага** (сервис, слой, правка конфига, форматирование), а не одним большим коммитом в конце; несвязанные изменения разносятся по отдельным коммитам своего типа.
10. **Чистота репозитория и архитектуры — постоянная обязанность.** Никаких мёртвых файлов, черновиков, дублей и «times.md» в корне; каждый файл лежит строго на своём месте по структуре [.agents/12-deployment.md](.agents/12-deployment.md) §1; слои и границы контекстов не размываются (проверяй перед каждым коммитом: `git status` чистый, структура каталогов соответствует документации, ничего лишнего не закоммичено).
11. **Агентам разрешены любые CLI-команды** (сборка, запуск, docker, gh, диагностика — без ограничений по списку) и доступен **chrome-devtools MCP**. Фронтенд не делается вслепую: после значимой правки UI агент обязан открыть страницу в браузере через chrome-devtools MCP, **сделать скриншот и посмотреть на результат** (плюс консоль на ошибки), сверить с дизайн-системой и анти-чеклистом [.agents/17-frontend.md](.agents/17-frontend.md), и только затем считать задачу выполненной.
