# 08. Эпистемический слой: консенсус, противоречия, пробелы, эксперты

Батч/инкрементальный контур (kmap-epistemic, Go). Не в горячем пути: результаты предрассчитаны и читаются search'ем из `epi.*`. Триггер — события `cluster-dirty`, страховка — ночной полный пересчёт с drift-детектором.

## 1. Классы сопоставимости условий (condition_class)

Урок E3.0: сравнивать утверждения можно только при сопоставимых условиях. Канонизация `conditions` фактов в дискретные классы:

| Условие | Полосы (пример конфига `configs/base/epistemic-bands.yml`) |
|---|---|
| temperature_c | <0, 0–25, 25–60, 60–90, 90–200, 200–700, >700 |
| current_density_a_m2 | <150, 150–250, 250–400, >400 |
| climate | cold / temperate / unspecified |
| medium | sulfate / chloride / mixed / unspecified |
| material_class | nickel_ore / copper_ore / matte / solution / … |
| scale | lab / pilot / industrial |

`condition_class = {условие: полоса}` только по **значимым** для параметра условиям (список значимых условий на параметр — в `parameter_defs.notes`, дефолт — все присутствующие). Ключ кластера: `ckey = sha256(subject_id, parameter_id | predicate+object_id, canonical(condition_class))`. Отсутствующее условие = `unspecified` — отдельная полоса (не смешиваем «не указано» и «указано другое»).

## 2. Консенсус (numeric-кластеры)

Для кластера интервалов в SI `[aᵢ, bᵢ]` с весами `wᵢ = extraction_confidence × source_reliability × freshness`:

- **overlap_index** — обобщённый Жаккар: `|⋂ᵢ Iᵢ| / |⋃ᵢ Iᵢ|` (точечные значения расширяются на ±2% для устойчивости);
- **agreed_range** — взвешенная медиана границ (робастна к выбросам);
- **verdict**: `insufficient` (<2 независимых документов) → `consensus` (OI ≥ 0.5 ∧ ≥3 документа) → `majority` (≥60% интервалов пересекают agreed_range) → иначе `split`;
- `stats`: источники ru/foreign, годы, распределение по scale — идёт в ответ («7 источников, 2 российских, 2018–2025»).

Claim-кластеры (качественные): консенсус по знаку `polarity` с теми же весами; `split` при доле меньшинства >0.3.

## 3. Противоречия: candidate → judge → expert

**Генерация кандидатов (детерминированная), только внутри одного кластера (= одинаковый condition_class):**
- `range_disjoint`: OI пары = 0 и зазор > 10% от объединения;
- `polarity_conflict`: claims с противоположной polarity об одной паре (subject, object);
- `applicability`: `applicable_for` vs `not_applicable` одной технологии к одному контексту;
- `result_mismatch`: эксперименты с одинаковым condition_class и расхождением результата по свойству > 2× межквартильного размаха кластера.

Пары из **разных** condition_class кандидатами не являются — они отображаются в UI как «различие контекста» (см. §6), не как конфликт. Это главный фикс precision 0.42 → 0.83 (E3.1).

**LLM-judge (задача `judge_contradiction`, Qwen3-32B, батч):** вход — оба факта с цитатами, условиями, метаданными источников; выход по схеме: `{is_contradiction: bool, confidence, likely_confounders: [...], severity: 0..1, rationale}`. Правило по умолчанию в промпте: **сомневаешься — не противоречие** (управляем precision, recall жертвуем осознанно). Экономия: результат кэшируется по паре fact_id (kmap-llm cache), пересчёт только при изменении членов пары.

**Статусная модель:** `suspected → judge_confirmed | judge_rejected`; эксперт в админке может `expert_confirmed | expert_rejected | resolved` (с комментарием — в fact_history). На витрину (ответы, граф — красные рёбра) попадают только `judge_confirmed+`. `CONTRADICTS`-ребро в `kg.edges` создаётся/удаляется этим контуром.

Факты-участники подтверждённого противоречия получают `validation_status='contradicted'` (виден и в ранжировании — понижение).

## 4. Пробелы и покрытие (coverage/gap)

**Пространство ячеек:** `домен × материал × процесс × [ключевое условие]`. Против комбинаторного взрыва — **co-occurrence prior** (E5.1): ячейка существует, если пара (материал, процесс) встречена хотя бы в одном документе корпуса ИЛИ объявлена в справочнике применимости (seed). Условие добавляет измерение только для приоритетных осей (climate, medium).

**Счётчики ячейки:** docs, experiments, facts, validated_facts, experts, ru_docs/foreign_docs, last_source_year.

**Score (0–100), перцентильная нормировка внутри домена:**
```
score = 100 × (0.30×pct(docs) + 0.30×pct(experiments) + 0.15×pct(experts)
             + 0.15×recency + 0.10×pct(validated_facts))
recency = exp(-0.15 × (текущий_год − last_source_year))
```
`gap_flag = score < P25(домена) ∨ experiments = 0`; `gap_reasons ⊆ {no_experiments, no_ru_practice, no_foreign_practice, stale(>7 лет), low_validation, no_experts, contradictory}` (contradictory — если в ячейке ≥1 подтверждённое противоречие). Компоненты score хранятся и показываются (никакого «магического числа»).

**Выходы:** (а) heatmap-дашборд руководителя (матрица материал×процесс, drill-down в ячейку → документы/эксперименты/эксперты/причины); (б) панель Gaps в ответе на запрос: ячейки плана с gap_flag + **рекомендации смежных областей** — соседние ячейки без gap_flag по осям (тот же материал/другой процесс; тот же процесс/другой материал; та же пара/другой климат), отранжированные по score — это детерминированная реализация «похожих кейсов из смежных областей» из ТЗ.

## 5. Профили экспертов (H-C, институциональная память)

Источники: перечень сотрудников/лабораторий (seed), `AUTHORED`-связи из документов, каталог экспериментов (исполнители), `VALIDATED_BY` (экспертные действия в системе).

```
weight(person, entity) = Σ_docs role_weight × doc_relevance × exp(-0.15 × (now − doc_year))
  role_weight: автор отчёта 1.0, соавтор 0.7, исполнитель эксперимента 1.0, валидатор 0.4
  doc_relevance: доля фактов документа, привязанных к entity (0..1)
```
Evidence-цепочка (документы, эксперименты, годы) хранится в `epi.expert_topics.evidence` — ответ «кто работал над X» всегда раскрывается до первоисточников. Приватность: доступ к профилям — по ролям (researcher видит имена и лаборатории; external_partner — только лаборатории без ФИО); в JSON-LD экспорт персональные данные не включаются.

## 6. UI-представление слоя

- Вкладка **Противоречия**: карточки «Источник A утверждает … [цитата] / Источник B … [цитата] / Вероятная причина: различие плотности тока (judge, confidence 0.86) / статус»; кнопки эксперта: подтвердить/отклонить/разрешить.
- «Различия контекста» — отдельная секция той же вкладки (не пугаем ложными конфликтами, но показываем разброс).
- Вкладка **Пробелы**: причины + смежные рекомендации + эксперты соседних ячеек.
- **Heatmap** (руководитель): цвет = score, бейджи причин; фильтры по домену/оси; экспорт CSV.
- Граф: `CONTRADICTS` — красные рёбра; вес ребра = подтверждения; прозрачность = confidence; серые узлы = pending_review (правила RFC §17.5 сохранены).

## 7. Инкрементальность (KR-7 «новые связи сами»)

`facts.committed{cluster_keys}` → upsert членств → пересчёт консенсуса кластера → ре-генерация кандидатов только для новых пар (новый факт × существующие члены) → judge-батч по расписанию (каждые 5 мин либо ≥20 пар) → обновление coverage-ячеек, задетых документом → обновление expert_topics авторов документа → событие `epistemic.v1.updated` (инвалидация кэша ответов, notify Phase 2). Полный пересчёт (ночь): идемпотентная переборка всех dirty=false кластеров с сверкой контрольных сумм; расхождение >1% — алерт (дрейф инкрементальной логики).
