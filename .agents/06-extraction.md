# 06. Извлечение знаний: детерминированное числовое ядро + LLM-контур

Ядро доверия системы (гипотеза H-A). Правило разделения: **значения, единицы, операторы — только детерминированный код; смысловая привязка и качественные связи — LLM под схемой с верификацией.**

## 1. Numeric core (детерминированный экстрактор)

Пакет `numcore` (Python, версионируется `numcore-X.Y.Z`, идёт в `extractor_version` каждого факта; покрыт golden-тестами: 150+ размеченных примеров в `eval/numeric_gold.yml`).

### 1.1. Конвейер

```
текст блока/ячейки таблицы
 → нормализация записи (юникод: – − —, ×10ⁿ, e-notation, десятичная запятая→точка,
    неразрывные пробелы, надстрочные ²³, «º»→«°»)
 → токенизация кандидатов (регекс-семейства с именованными группами)
 → парс оператора и значения (min/max)
 → резолв единицы через реестр kg.units (+Pint) → SI
 → извлечение локальных условий (co-located conditions в том же предложении/строке)
 → sanity-проверки (parameter_defs) → confidence, флаги
 → NumericCandidate {span, quote, operator, vmin, vmax, unit_code, vmin_si, vmax_si, ...}
```

### 1.2. Регекс-семейства (76 паттернов, основные классы)

| Класс | Примеры входа | operator / значения |
|---|---|---|
| точное значение | `0.8 м/с`, `250 мг/л`, `pH 4,5` | `eq`, vmin=vmax |
| диапазон тире | `60–80 °C`, `0.6-0.9 м/с`, `200…300 мг/л` | `range` |
| диапазон словами | `от 0.6 до 0.9 м/с`, `в пределах 60—80` | `range` |
| верхняя граница | `≤1000 мг/дм³`, `не более 300`, `до 300 мг/л`, `<200` | `lte`/`lt`/`to` |
| нижняя граница | `≥100`, `не менее 100`, `от 100 т/сут`, `свыше 0.7`, `выше 0.7` | `gte`/`gt`/`from` |
| допуск | `65±2 °C`, `0.8 ± 0.05 м/с` | `pm` → range |
| приближение | `около 100`, `порядка 100`, `~100`, `примерно` | `approx` |
| научная нотация | `3·10⁻² моль/л`, `1.2e5 Па`, `5×10^3` | по контексту |
| проценты и доли | `95%`, `4.2 мас.%`, `об.%`, `г/т`, `ppm`, `‰` | по контексту |
| кратные изменения | `в 1.5 раза выше`, `на 4.2%` (дельта свойств) | `delta`-флаг в conditions |
| экономика | `120 руб/т`, `$2.4 млн CAPEX`, `кВт·ч/т` | relation='economic' |

Ложные срабатывания отсекаются стоп-контекстами: номера рисунков/таблиц/формул/патентов/ГОСТов, годы (4-значные 19xx/20xx без единицы), нумерация списков, DOI, «стр. 12».

### 1.3. Реестр единиц

`kg.units` (~90 записей на старте) + Pint UnitRegistry с кастомным файлом определений (`units_kmap.txt`): все RU-написания и варианты (`мг/л|мг/дм3|мг/дм³|mg/L`, `А/дм²|A/dm2`, `т/сут|t/day`, `кВт·ч/т`, `мас.%|wt.%`, `г/т|g/t|ppm`). Резолвинг: точный матч по `names[]` → Pint-парс как fallback → неуспех = `unit_code IS NULL` → факт получает `needs_unit_review` и **исключается из числового поиска** (единицы не угадываем — KR-1). Перевод в SI: `si = value*si_factor + si_offset` (°C→K с offset 273.15); денежные единицы не конвертируются между валютами — хранится `unit_code` валюты и год (дефляторы вне скоупа MVP).

**Реализация MVP:** авторитетный реестр — data-driven `services/extract/src/extract/units.yml` (70 единиц по 16 измерениям: концентрации, температура, давление, плотность тока, расходы, удельная энергия, pH, экономика; каждая запись — `code`, `names[]`, `dimension`, `si_factor`, `si_offset`, привязка к параметру и SI-диапазон правдоподобия). Он зеркалит `kg.units` (тот же набор кодов сеидится в `db/seeds/demo.sql`), поэтому `unit_code` фактов всегда резолвится по FK. Резолвинг реализован как точный матч по `names[]` (lower-case, longest-first); Pint-парс как fallback для неизвестных токенов **сознательно отложен** — на precision-первом горячем пути неизвестная единица не угадывается, а факт просто не извлекается (KR-1 важнее recall). Возврат Pint-fallback — при появлении единиц вне доменного реестра.

### 1.4. Sanity-проверки

По `kg.parameter_defs`: (а) согласованность размерности параметра и единицы (температуру нельзя измерить в мг/л — такой кандидат отклоняется с логом); (б) диапазон правдоподобия (temperature процессов: −60…1700 °C; pH 0…14; скорость потока 0…20 м/с; концентрации ≤ насыщения класса) — вне диапазона → `pending_review`, confidence×0.5, алерт в очередь ревью. Проверки не удаляют данные — только маркируют.

### 1.5. Привязка «число → сущность» (гибрид)

Кому принадлежит `0.8 м/с`? Каскад:
1. **Таблицы**: колоночная семантика (заголовок → parameter, строка → experiment) — детерминированно, покрывает каталог экспериментов.
2. **Текст, паттерны**: «<параметр> составлял/равнялся/поддерживался на уровне <значение>», «при <значение>» рядом с упоминанием параметра в том же предложении (окно ±1 предложение) — детерминированно.
3. **Остаток** — LLM-задача `bind_numbers`: вход — предложение + список найденных numcore-кандидатов (span'ы) + кандидаты сущностей; выход — привязки `{candidate_span → parameter_slug, subject_slug}`. **Инвариант: LLM не может изменить значение/единицу** — ответ принимается только если span в ответе байт-в-байт совпадает с кандидатом numcore; иначе привязка отклоняется (факт остаётся непривязанным, `orphan=true`, виден в админке).

Итоговый `extraction_method`: `deterministic` (1–2) | `hybrid` (3). Доля hybrid в замерах E1 — ~22% фактов.

## 2. LLM-контур извлечения

Все вызовы — через kmap-llm (schema-validated, см. [09-services.md](09-services.md) §7). Промпты версионируются в репо `prompts/<task>/<ver>.yml` (система+few-shots), ссылка на версию — в каждом артефакте.

### 2.1. Задачи и модели (матрица «экономия ресурсов»)

Фактический провайдер chat/synthesis — Yandex AI Studio (Responses API, `ai.api.cloud.yandex.net`, схема `Api-Key`, ключ+folder_id в `configs/secrets.yml`); эмбеддинги/rerank — DigitalOcean Gradient. Модели (`configs/base/llm-routes.yml`): `gpt-oss-20b/latest`, `gpt-oss-120b/latest`, `deepseek-v4-flash/latest`, `qwen3.6-35b-a3b/latest`.

| Задача | Модель по умолчанию | Эскалация | Когда эскалируем |
|---|---|---|---|
| `extract_entities` (сущности/связи/выводы из чанка) | `openai-gpt-oss-20b` (0.05/0.45, `reasoning_effort: low`, json_object) | `openai-gpt-oss-120b` (0.10/0.70) | valid-rate JSON < 0.9 по документу |
| `bind_numbers` | `openai-gpt-oss-20b` | `mimo-v2.5` (кросс-чек) | конфликт привязок |
| `parse_query` (QueryPlan) | `openai-gpt-oss-20b` | — | rule-fallback при недоступности |
| `synthesize_answer` | `deepseek-4-flash` (0.112/0.224, вывод до 65k, стриминг; json-mode недоступен → prompt-based JSON + repair) | `openai-gpt-oss-120b` (max_out 4k) | сбой/деградация |
| `judge_contradiction` | `openai-gpt-oss-120b` (reasoning, json_object) | `alibaba-qwen3-32b` → `qwen3.5-397b-a17b` (спорные) | батч, вне горячего пути |
| `suggest_aliases` (синонимы) | `mimo-v2.5` (0.105/0.28) | — | офлайн, статус pending |
| `literature_review` (длинный контекст) | `deepseek-4-flash` (65k) | `gemma-4-31B-it` (256k), `nvidia-nemotron-3-super-120b` (1M) | документы не влезают |

Правила kmap-llm для этого провайдера: отбрасывать `reasoning_content` из ответов (gpt-oss/mimo) и учитывать, что reasoning-токены биллятся как output (`reasoning_effort: low` для дешёвых задач); **allowlist только open-weight** — проприетарные `openai-gpt-5*`/`anthropic-*` из каталога DO запрещены правилами хакатона.

Провайдеры (OpenAI-совместимые): (а) **DigitalOcean Gradient — API владельца, выдан и проверен 02.07, рабочий дефолт**; (б) реестр моделей организаторов (доступ с утра 03.07 — сравним по ценам/качеству и переприоритизируем); (в) on-prem vLLM с Qwen3 (Apache 2.0) — офлайн-резерв; (г) routerai.ru — внешний резерв. Всё разрешено правилами хакатона; точные ID моделей — конфиг (`configs/<env>/llm-routes.yml`), не код.

### 2.2. `extract_entities` — контракт

Вход: чанк + section_path + типы сущностей/связей + уже известные canonical-слаги упомянутых сущностей (context hints из resolution-префетча). Выход (JSON Schema, `strict`):

```json
{
  "entities": [{"type": "material", "name": "никель", "name_en": "nickel",
                "span": [123,129], "confidence": 0.97}],
  "relations": [{"src": "EXP-014", "src_type": "experiment", "rel": "USES_MATERIAL",
                 "dst": "никель", "dst_type": "material", "span": [100,140], "confidence": 0.9}],
  "claims": [{"subject": "скорость циркуляции католита", "predicate": "improves",
              "object": "равномерность осаждения никеля", "polarity": 1,
              "conditions": {"process": "электроэкстракция"},
              "quote": "...", "span": [..], "confidence": 0.83}]
}
```

Правила промпта (санитарные): «не выдумывай; факт только из текста; каждый элемент со span; числа НЕ извлекай (их извлекает отдельный контур); если не уверен — не включай». Пост-валидация: span существует и содержит упомянутые токены; типы из enum; отклонённые элементы логируются с причиной (метрика `llm_reject_rate`).

### 2.3. Условия (conditions) фактов

Из локального контекста (предложение/строка таблицы/секция) собирается словарь условий: `{"temperature_c": [60,80], "current_density_a_m2": 250, "climate": "cold", "material_class": "nickel_ore", "medium": "sulfate"}`. Числовые условия извлекает numcore, категориальные — детерминированные словари + LLM-дозаполнение (тоже под схемой). `condition_hash = sha256(canonical_json(conditions))`. Условия — фундамент epistemic-слоя (классы сопоставимости, [08-epistemic.md](08-epistemic.md) §2).

### 2.4. Geography-резолвер

`ru | foreign | global | unknown` для документа и переопределение на факте, если в тексте явное указание («в отечественной практике», «на заводе Х (Финляндия)»): словарь маркеров + страны → `kg.entities(etype='geography')`. Документ-уровень: издание/язык/аффилиации; RU-язык сам по себе не означает `ru`-практику (перевод зарубежного обзора) — приоритет у аффилиаций и явных маркеров.

## 3. Чанкинг

- Текст: по секциям (section_path из DocIR), цель 800–1200 токенов, overlap 120; заголовок секции добавляется в начало текста чанка (улучшает и FTS, и вектор).
- Таблицы: каждая строка → `table_row`-чанк (текстовое представление по шаблону «Эксперимент EXP-014: материал — никель; …»), вся таблица → `table`-чанк с caption.
- Формулы/подписи рисунков — отдельные чанки `kind='caption'` (часто содержат ключевые числа).

## 4. Качество и контроль

- **Golden-тесты numcore** в CI: любой PR, меняющий грамматику, гоняет 150+ примеров; регресс = красный билд.
- Метрики по документу в bundle.quality: `nc_count`, `nc_suspect_rate`, `llm_valid_rate`, `orphan_rate` — алерты при аномалиях (например, отчёт без единого числа — вероятно, битый парс).
- Пер-извлечение confidence: numcore — из класса паттерна и sanity-результата (0.95–0.99 базово); LLM — самооценка×калибровочный коэффициент задачи (замер на gold).
- Целевые значения: precision чисел ≥0.98, recall ≥0.94, unit accuracy ≥0.99, entity F1 ≥0.85, relation F1 ≥0.75 ([13-evaluation.md](13-evaluation.md)).

## 5. Статус реализации MVP (Python-конвейер)

Реализовано и проверено в Docker (`services/{parse,extract,embed,eval}`, событийная связка через NATS):

- **parse** (`services/parse`): консьюмер `kmap.doc.v1.registered` → извлечение текста по magic-bytes (PDF через `pypdf`, DOCX через `python-docx`, txt/md напрямую; полный Docling с OCR — осознанно отложен, чтобы образ оставался `slim` без torch) → DocIR (`docir/1`, блоки-абзацы, `full_text`) в MinIO → `kmap.doc.v1.parsed`.
- **extract** (`services/extract`): консьюмер `kmap.doc.v1.parsed` → детерминированный numcore (грамматика чисел/операторов/единиц + SI + `condition_hash`) → ExtractionBundle в MinIO → `kmap.doc.v1.extracted`; catalog-консьюмер коммитит факты в PG. LLM-извлечение сущностей/связей — следующий шаг (deterministic-ядро работает автономно, KR-1 закрыт).
- **embed** (`services/embed`): gRPC `EmbedService` (Embed 1024d + Rerank), backend `deterministic` (offline-хэш, дефолт) | `remote` (bge-m3 через DO Gradient); Python-стабы генерируются локально в `services/embed/src/embed/gen`, `contracts/` не трогается. Векторный канал search подключается на стороне Go по мере готовности.
- **eval** (`services/eval`): harness прогоняет эталонные Q1–Q6 через `/v1/ask`, проверяет numeric guard; golden-тесты numcore (`services/extract/tests`, 19 кейсов) — в `task py-test`. Текущий прогон: 6/6, `hallucinated_numbers_rate = 0`.

Живой сквозной путь подтверждён: загрузка текстового документа через `/v1/documents` → parse → extract → catalog → факты в PostgreSQL.
