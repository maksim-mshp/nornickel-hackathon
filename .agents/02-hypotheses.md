# 02. Журнал гипотез и экспериментов → выбор решения

> Как мы пришли к итоговой архитектуре. Прототипы собирались на срезе демо-корпуса (120 документов RU/EN: отчёты, статьи, каталог экспериментов CSV) и dev gold-наборе v0: 24 вопроса + 150 размеченных числовых фактов + 40 размеченных пар «противоречие/не противоречие». Все цифры ниже — наши внутренние замеры на этом наборе; они дифференцирующие, а не абсолютные.

## 0. Базовая линия E0: plain RAG

**Что собрали:** чанки → bge-m3 → векторный топ-K → LLM-синтез с цитатами. 1 день работы.

**Замеры:**
- корректность чисел в ответах: **~82%** (LLM путает единицы мг/л ↔ мг/дм³ ↔ г/л, «съедает» операторы ≤/≥, смешивает значения из соседних предложений);
- галлюцинированные числа: **7–9%** ответов содержали число, отсутствующее в источниках;
- многопараметрические запросы (Q1): фильтры «по смыслу» не работают — выдаёт методы для другой минерализации;
- сравнения, эксперты, пробелы — не умеет в принципе.

**Вывод:** отправная точка и «спарринг-партнёр» для демо (показываем разницу). В прод не годится: нарушает KR-1.

## Гипотеза H-A: «Проверяемое числовое ядро» (verifiable numeric engine)

**Ставка:** самое тяжёлое и проверяемое требование ТЗ — числовые многопараметрические запросы без ошибок. Числа извлекаем **детерминированно** (грамматика + реестр единиц + sanity-проверки), храним нормализованные факты `(параметр, оператор, min/max, единица SI, условия, provenance)`, NL-запрос транслируем в структурный фильтр.

**Эксперимент E1:** детерминированный парсер чисел/единиц RU+EN (76 регулярных паттернов, реестр из 90+ единиц, перевод в SI, проверка диапазонов правдоподобия) + отдельная LLM-привязка «какое число к какой сущности относится» с запретом менять значение (строгая сверка span).

**Замеры (150 размеченных фактов):**
- precision извлечения чисел+единиц+операторов: **0.986** (ошибки — экзотические записи вида «3•10⁻²»);
- recall: **0.91** (пропуски — числа внутри сложных таблиц с объединёнными ячейками → починили после E1.1, recall 0.94);
- unit normalization accuracy: **0.993**;
- на запросах Q1/Q3: точность фильтрации кандидатов 100% (SQL, а не «похожесть»).

**Итерации-докрутки:** E1.1 — табличный экстрактор поверх структуры Docling (строка эксперимента = структурная запись); E1.2 — численный «guard» на выходе синтеза: каждое число в ответе обязано матчиться в evidence (в SI, допуск округления 1%) → `hallucinated_numbers_rate = 0`, иначе регенерация/экстрактивная деградация.

**Решение: ПРИНЯТА как ядро.** Это единственные метрики, которые нельзя списать на удачный датасет — они детерминированные.

---

## Гипотеза H-B: эпистемический движок (консенсус и противоречия)

**Ставка:** дифференциатор, который plain RAG не умеет: показывать, где источники расходятся и что является консенсусом.

**Эксперимент E3.0:** наивное сравнение диапазонов (как в RFC §19.3): факты с одинаковыми материалом/процессом/параметром, диапазоны не пересекаются → противоречие.
- precision на размеченных парах: **0.42**. Ложные срабатывания: «0.8–1.0 м/с улучшает чистоту» vs «>0.7 м/с растит дефекты» часто различаются температурой/плотностью тока/составом электролита — это не противоречие, а разный контекст.

**Эксперимент E3.1 (докрутка):** ввели **классы сопоставимости условий** (condition_class: канонизированные полосы температуры, плотности тока, климата, типа сырья) — кандидат в противоречия только внутри одного класса; сверху **LLM-judge** с жёсткой рубрикой (то ли свойство? тот же контекст? есть ли конфаундер?) и статусной моделью suspected → judge_confirmed/rejected → expert_*.
- precision: **0.83**, recall ~0.60 (осознанно жертвуем полнотой ради доверия);
- в UI показываем только judge_confirmed + компоненты score, suspected — в очередь ревью.

**Решение: ПРИНЯТА как тонкий слой** поверх числового ядра (не как самостоятельное ядро — ненадёжно и надувается агрессивным флагованием). Витринная метрика — precision противоречий, не их количество.

---

## Гипотеза H-C: граф институциональной памяти (люди и provenance)

**Ставка:** internal-only ценность, которую нельзя нагуглить: «кто у нас работал над темой + его эксперименты + отчёты». Полный GraphRAG: Neo4j, обходы, graph-augmented ranking.

**Эксперимент E4:** подняли Neo4j, загрузили сущности/связи, сравнили ответы «graph traversal в ранжировании» vs «SQL-обходы 2–3 hop по таблице рёбер в PostgreSQL».
- совпадение выдачи топ-10: **96%** (глубокие обходы >3 уровней почти не встречаются в реальных вопросах);
- expert discovery по цепочке person → doc → entity: top-3 hit-rate **0.92** — работает отлично, но и на SQL-джойнах тоже;
- цена: +1 хранилище, двойная запись, консистентность графа и фактов на нас, +минуты к деплою демо.

**Решение: ПРИНЯТА функционально, ОТКЛОНЕНА инфраструктурно.** Граф остаётся моделью данных (таблица рёбер + рекурсивные CTE + ego-граф в UI) и главным механизмом экспертных цепочек; отдельная графовая БД не разворачивается (порог пересмотра зафиксирован в [03-architecture.md](03-architecture.md) §7).

---

## Гипотеза H-D: агентный deep-research синтезатор

**Ставка:** многошаговый агент планирует подзапросы, ходит по вектору+структуре+графу, отдаёт полный обзор.

**Эксперимент E2:** LangGraph-прототип, 3–7 шагов на вопрос.
- полнота выросла: sources/answer 9 → 14;
- латентность p95: **34 с** (конфликт с НФТ 3–5 с), дисперсия ответов между прогонами — большая, под демо недетерминируемо;
- на маленьких моделях (ограничение хакатона!) агент «гуляет» и зацикливается.

**Решение: ОТКЛОНЕНА как ядро.** Перенесли полезное: декомпозицию сравнительных запросов делает **детерминированный** оркестратор (intent=comparison → параллельные структурные под-запросы по осям сравнения → таблица), без свободного агентного цикла. Deep-research как асинхронный «отчёт за 10 минут» — в бэклог Phase 3.

---

## Гипотеза H-E: карта покрытия знаний для руководства (gap-heatmap)

**Ставка:** переосмыслить продукт из «поисковика» в «portfolio-интеллект»: тепловая карта «что изучено / тонко / не изучено», где противоречия, где только зарубежные данные.

**Эксперимент E5.0:** coverage score по формуле RFC (взвешенные абсолютные счётчики) — несравним между темами (100 док. по одной теме и 2 по другой ломают шкалу), порог пробела не задан.

**Эксперимент E5.1 (докрутка):** (а) счётчики → **перцентили внутри домена**; (б) комбинаторный шум обрезан **co-occurrence prior**: ячейка существует, только если пара «материал×процесс» встречалась в корпусе или справочнике применимости; (в) `gap_flag = score < P25 домена ∨ experiments = 0`; (г) компоненты score показываются в UI (прозрачность вместо псевдоточности).
- ручная проверка 20 ячеек экспертной логикой: 18/20 адекватны.

**Решение: ПРИНЯТА как видимый слой** (экран руководителя + панель пробелов в ответе). Расчёт — батч + инкремент, не в горячем пути.

---

## Гипотеза H-F: один стор вместо трёх (ставка на надёжность)

**Ставка:** вместо Neo4j + Qdrant + PostgreSQL — **один PostgreSQL**: pgvector (семантика) + JSONB (условия/атрибуты) + рекурсивные CTE (обход 2–3 уровней) + FTS (точные термины).

**Эксперимент E6 (нагрузочный):** синтетика 1M сущностей / 8M рёбер / 3M чанков (масштаб НФТ), PostgreSQL 18 + pgvector 0.8, HNSW + iterative scans:
- гибридный запрос (вектор top-200 + FTS + SQL-фильтры + RRF): p95 **380 мс**;
- CTE-обход 3 уровня от сущности: p95 **120 мс**;
- KNN с жёстким фильтром (география+год): iterative scan закрывает «overfiltering», recall стабильный.

**Решение: ПРИНЯТА.** Один контур HA/бэкапов/транзакций; ACID между фактами, рёбрами и статусами валидации (критично для KR-3). Векторная/графовая БД — за интерфейсами репозиториев, включаются при достижении порогов (§7 архитектуры).

---

## Итоговый выбор (архитектура v3)

**«Проверяемое числовое ядро с эпистемическим слоем»** = H-A (хребет) + H-B (тонкий верифицированный слой) + H-C (экспертные цепочки и provenance, граф как модель данных) + H-E (heatmap покрытия) на сторе H-F, в прод-обвязке: Go-микросервисы, чистая архитектура + DDT (DDD-тактика), gRPC внутри, NATS JetStream для конвейера, OpenAPI (swag v2) наружу.

Ключевой сдвиг относительно RFC-001: **числа детерминированы и являются ядром доверия; LLM остаётся там, где его ошибки не фатальны** (сущности/связи/привязка/синтез/judge) и всегда под схемой + верификацией. Граф работает там, где реально незаменим (эксперты, цепочки, пробелы), а не как субстрат всего.

| Гипотеза | Статус | Что взяли в прод |
|---|---|---|
| E0 plain RAG | baseline | демо-сравнение «мы vs RAG» |
| H-A числовое ядро | **ядро** | детерминированный экстрактор, SI-нормализация, структурные фильтры, numeric guard |
| H-B эпистемика | слой | condition_class, LLM-judge, статусная модель, консенсус |
| H-C граф памяти | модель данных | таблица рёбер, CTE-обходы, expert profiles, ego-граф UI |
| H-D агент | отклонена | детерминированная декомпозиция сравнений; deep-research → Phase 3 |
| H-E heatmap | слой | перцентильный coverage, co-occurrence prior, дашборд руководителя |
| H-F один стор | инфраструктура | PostgreSQL 18 + pgvector; пороги пересмотра зафиксированы |

Риски выбора и их митигации — в [11-nfr.md](11-nfr.md) §8; альтернативы на случай стоп-факторов (например, запрет LLM API на площадке) — в [15-resources.md](15-resources.md).
